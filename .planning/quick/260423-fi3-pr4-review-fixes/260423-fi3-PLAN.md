  <name>Task 2: PhotographerContext refreshData 에서 userId 의존성 분리 (이중 페치/UI 블랭크 제거)</name>
  <files>app/src/contexts/PhotographerContext.tsx</files>
  <action>
현재 refreshData useCallback 의존성에 `userId` 가 포함돼 있어, 비로그인 상태로
초기 마운트 → 로그인 완료 시 userId 변경 → refreshData 재생성 → useEffect 재실행
→ `setLoading(true)` 재발화 + public data (photographers/photoPosts/players/...)
재페치가 발생한다. 결과적으로 로그인 직후 HomeScreen 등이 잠시 로딩/블랭크
상태로 깜빡이고 불필요한 네트워크 round-trip 이 생긴다.

리팩터 방침 (권장안 — 관심사 분리):

1. `refreshData` 는 **public data 전용** 으로 축소한다.
   - 내부의 `if (userId)` 블록 (likesRes/followsRes 페치 + setPhotoLikedIds /
     setFollowedPgIds) 및 `else` 블록(두 Set 을 비움)을 **제거**한다.
   - useCallback 의존성 배열에서 `userId` 를 제거한다 → `useCallback(..., [])`.
   - 기존 try/catch/finally 구조, setLoading(true)/setLoading(false) 는 유지한다
     (초기 public data 로딩 상태 표현 유지).

2. user-scoped 복원 전용 effect 를 신규 추가한다.
   - 새 useCallback `refreshUserScoped` 를 만든다:
     - userId 가 null 이면 `setPhotoLikedIds(new Set()); setFollowedPgIds(new Set())`
       를 호출하고 early return.
     - userId 가 있으면 기존의 `Promise.all([fetchUserPhotoLikes(userId),
       fetchUserFollows(userId)])` 로직을 옮기고 결과를 set 한다.
     - **setLoading 은 호출하지 않는다** (전역 로딩 토글 금지 — public data 로딩과
       혼동 방지).
     - 의존성: `[userId]`.
   - `useEffect(() => { void refreshUserScoped(); }, [refreshUserScoped]);` 추가.

3. 기존 `useEffect(() => { void refreshData(); }, [refreshData]);` 는 그대로 두되,
   이제 refreshData 의존성이 빈 배열이 되어 **mount 시 1 회만** 실행된다.

4. 컨텍스트 value / useMemo 의존성 배열은 함수 참조가 안정화되므로 기존과 동일하게
   유지한다 (refreshData 는 여전히 노출). refreshUserScoped 는 내부 전용이므로
   value 에 포함하지 않는다.

주의사항:
- 기존 [WR-02] 주석 (개별 API tuple 에서 error 관찰) 은 public results.forEach
  에만 남기고, user-scoped 쪽에도 likesRes.error / followsRes.error `console.warn`
  은 그대로 유지한다.
- `catch (e)` 의 `console.error('[PhotographerContext] refreshData unhandled', e)`
  로그 prefix 는 refreshData 쪽은 기존 유지, 새 refreshUserScoped 쪽은
  `'[PhotographerContext] refreshUserScoped unhandled'` 로 구분한다.
- 수동 `refreshData()` 호출부 (pull-to-refresh 등 consumer 에서 호출하는 경우)는
  public data 만 새로고침하게 되므로 동작 차이 발생. 이는 의도된 동작 (좋아요/
  팔로우는 userId 변경 시점에만 복원하면 충분). 리뷰어가 별도 공지하지 않았고,
  현재 consumer grep 시 명시적 pull-to-refresh 호출 위치가 없다면 그대로 유지.
  있으면 주석으로 명시.
