**수정 방향 (04-REVIEW.md 옵션 A — is_photographer 플래그 활용):**

AuthContext 의 `user.is_photographer: boolean` 은 로그인 시 `public.users` row 에서 직접 로드되므로, 앱이 인증된 사용자로 cold start 할 때 이미 load 되어 있다. 이를 초기 state 로 사용하면 camera → aperture 전환이 사라진다.

1. **초기 state**를 `user?.is_photographer` 기반으로 설정:
   ```typescript
   const [studioState, setStudioState] = useState<StudioTabState>(
     user?.is_photographer ? 'approved' : 'no_app'
   );
   ```
   `useState` 의 lazy initializer 는 첫 렌더에만 실행되므로 `user?.is_photographer` 가 이미 설정되어 있으면 'approved' 로 시작, 아직 session 이 load 전이면 'no_app' — 이 경우 사용자가 비로그인 또는 아직 profile 미로드 상태이므로 camera 라벨이 올바르다.

2. **user?.id 변경 시에도 is_photographer 를 반영** — useEffect 내부에서 fetch 전에 optimistic baseline 을 설정:
   ```typescript
   useEffect(() => {
     if (!user?.id) {
       setStudioState('no_app');
       return;
     }
     // 로그인 이벤트 직후 재 마운트 시 is_photographer 를 동기 초기값으로 사용
     if (user.is_photographer) {
       setStudioState((prev) => (prev === 'no_app' ? 'approved' : prev));
     }
     let cancelled = false;
     photographerApi.fetchMyPhotographerApplication(user.id)
       .then(...)
       .catch(() => {
         // 네트워크 장애 시 is_photographer 가 true 면 현재 state 유지 (downgrade 차단)
         if (cancelled) return;
         if (!user.is_photographer) setStudioState('no_app');
       });
     return () => { cancelled = true; };
   }, [user?.id, user?.is_photographer]);
   ```

3. **fetch 성공 시 분기 로직은 그대로 유지**:
   - `res.data == null` && `user.is_photographer == true` 인 모순 케이스는 드물지만 발생 가능 (e.g. users.is_photographer 는 true 인데 application row 가 없는 경우). 이 때는 `approved` 유지 (application 은 승인 후 archived 되어도 photographers row 로 판단 가능). 단 application 이 있으면 그 status 를 신뢰 (pending/rejected/approved).

   구체 코드:
   ```typescript
   .then((res) => {
     if (cancelled) return;
     if (!res.data) {
       setStudioState(user.is_photographer ? 'approved' : 'no_app');
     } else if (res.data.status === 'pending') setStudioState('pending');
     else if (res.data.status === 'rejected') setStudioState('rejected');
     else setStudioState('approved');
   })
   ```

```sql
-- 여러 행을 INSERT 할 수 있었음. fetchMyPhotographerApplication 이 order by
-- created_at desc limit 1 로 "최신만" 가져오는 방식은 숨김 효과일 뿐,
-- admin review 큐에는 중복이 누적되고 승인 트리거에서 notification 이 중복
-- 발송될 수 있었음 (WR-04 와 복합).
--
-- 해결: partial unique index 로 status='pending' 인 경우에만 user_id 유일성 강제.
-- approved/rejected 상태의 과거 이력은 중복 허용 (재신청 히스토리 보존).
-- ============================================================

CREATE UNIQUE INDEX IF NOT EXISTS photographer_apps_unique_pending
  ON public.photographer_applications (user_id)
  WHERE status = 'pending';

COMMENT ON INDEX public.photographer_apps_unique_pending IS
  'T-4-01 / HI-03: 사용자당 pending 신청은 최대 1건. 중복 INSERT 시 sqlstate 23505 unique_violation.';
```

**주의:**
- `IF NOT EXISTS` 사용 (멱등성).
- `ON public.photographer_applications` 풀 스키마 이름 명시.
- partial WHERE 절에 `status = 'pending'` 정확히 매칭 (리터럴).
- COMMENT 는 향후 schema browser 에서 의도를 파악할 수 있도록 추가.
