# Phase 3 — UI Design Contract (Community)

> Visual and interaction contract for the community Supabase migration. Generated by gsd-ui-researcher, verified by gsd-ui-checker. Phase 1 and 2 already established the app-wide design system; this contract captures **only the deltas** introduced by switching CommunityContext from mock to Supabase.

**Scope rule:** Where the existing theme tokens and components already cover a decision, this spec cites the token. New tokens or copy are only declared when Phase 3 introduces a state, interaction, or string the codebase does not yet have.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — custom React Native theme system (`app/src/styles/theme.ts`) |
| Preset | not applicable (Expo SDK 54 / React Native 0.81; shadcn is web-only) |
| Component library | none — handcrafted React Native components in `app/src/components/` |
| Icon library | `@expo/vector-icons` Ionicons (existing convention) |
| Font | System (iOS San Francisco / Android Roboto) — no custom font in v1 |
| Mode | Light only — dark mode is v2 (CONTEXT 03-OUT_OF_SCOPE) |
| Source of tokens | `app/src/styles/theme.ts` (re-exports `colors`, plus `spacing`, `fontSize`, `fontWeight`, `radius`, `shadow`, `layout`) |

**Source-of-truth rule:** All Phase 3 code MUST import tokens from `../../styles/theme`. No raw hex literals, no raw px integers for spacing/radius/font, no `StyleSheet` values that bypass the token system. If a needed value does not exist in the theme, it MUST be added to `theme.ts` first (and noted in Plan).

## Spacing Scale

The existing token scale (`app/src/styles/theme.ts → spacing`) is already a 4-point grid. Phase 3 reuses it as-is. **No new spacing tokens needed.**

| Token | Value | Usage in Phase 3 |
|-------|-------|------------------|
| `spacing.xs` | 4px | Icon-text gap inside post card stat row, badge inner gap |
| `spacing.sm` | 8px | Vertical gap between badges in card header, comment-action group spacing |
| `spacing.md` | 12px | Comment item vertical rhythm, write screen poll-option row gap |
| `spacing.lg` | 16px | Screen horizontal padding (header, list rows, post detail body), section padding |
| `spacing.xl` | 24px | Section separators between header / sort bar / list, FAB clearance |
| `spacing.xxl` | 32px | Empty/error state vertical padding (handled by `EmptyState` internal `paddingVertical: 60` constant — leave as-is) |

**Touch targets (a11y minimum 44pt):** Tap zones already comply via existing component dimensions (e.g. `headerBtn` is 36×36 wrapped in 56pt header row giving full 44pt+ tap area; FAB is 56×56). New Phase 3 interactive elements (retry button, dismiss-pagination-spinner, close-toast) MUST maintain minimum 44pt tap area, using the existing pattern of an inner icon centered in a wrapper view.

**Exceptions:** None for Phase 3.

## Typography

The existing token scale (`fontSize` + `fontWeight`) is already established and used app-wide. **Phase 3 reuses existing tokens — no new typography tokens.**

The full token table is in `app/src/styles/theme.ts`. Phase 3 uses the following subset (cite by token name, never px literals):

| Role | Token | Value | Weight token | Weight value | Used in Phase 3 for |
|------|-------|-------|--------------|--------------|---------------------|
| Screen header title | `fontSize.sectionTitle` | 21px | `fontWeight.heading` | 800 | "커뮤니티" header on `CommunityMainScreen` |
| Post detail title | `fontSize.price` | 18px | `fontWeight.heading` | 800 | Post title in `CommunityPostDetailScreen` (already used) |
| Body / comment text | `fontSize.body` | 15px | `fontWeight.body` (400-equivalent for input) | 600 | Post content, comment content, search input, write inputs |
| Section label / sub-heading | `fontSize.cardName` | 16px | `fontWeight.name` | 700 | "댓글 N" comment count header, write screen header title |
| Card title | `fontSize.body` | 15px | `fontWeight.name` | 700 | `CommunityPostCard.title` |
| Card content preview | `fontSize.meta` | 13px | `fontWeight.body` | 600 | `CommunityPostCard.content` snippet (line-height 18) |
| Meta / timestamp | `fontSize.micro` | 12px | `fontWeight.body` | 600 | "탈퇴한 사용자", timeAgo, poll percentage label, retry button text in EmptyState |
| Badge label | `fontSize.badge` | 11px | `fontWeight.heading` / `name` | 700–800 | Team badge, trending badge ("트렌딩"), poll badge ("투표") |

**Line height rules:**
- Body / post content: explicit `lineHeight: 22` (already in `CommunityPostDetailScreen.postContent` and existing convention) — keep as-is.
- Card content preview: explicit `lineHeight: 18` (already in `CommunityPostCard.content`) — keep as-is.
- Comments: `lineHeight: 20` (already in `commentText` style) — keep as-is.
- All new Phase 3 text additions (loading messages, error captions) MUST use `lineHeight ≈ size × 1.4` rounded to integer for body, `× 1.2` for short labels.

**No new font sizes or weights are permitted in Phase 3.** If a label visually requires a value outside the table, escalate to UI-checker rather than introducing literals.

## Color

Phase 3 uses the established palette. **No new color tokens needed.** The 60/30/10 split below cites exact `colors.*` references from `app/src/constants/colors.ts`.

| Role | Token | Value | Usage in Phase 3 |
|------|-------|-------|------------------|
| Dominant (≈60%) | `colors.background` | `#FFFFFF` | Screen backgrounds: `CommunityMainScreen`, `CommunityPostDetailScreen`, `CommunityWriteScreen`, `CommunitySearchScreen`, post card bg, input bar bg |
| Secondary (≈30%) | `colors.surface` | `#F5F7FA` | Team badge bg, post detail section divider (`borderBottomWidth: 8`), poll container bg, poll input bg, image thumb placeholder, recent search row hover, comment avatar bg, skeleton shimmer base |
| Borders/dividers | `colors.border` | `#E5E7EB` | All hairline dividers (sort bar bottom, list row separator, input bar top, poll option border, comment input border) |
| Accent (≈10%) | `colors.primary` | `#1B2A4A` Deep Navy | **Reserved-for list below** |
| Destructive | `colors.error` | `#EF4444` | Like-active (heart filled), trending badge text (`colors.trending` aliases this), error toast bg, "삭제" menu text in comment actions, char-count over-limit indicator, image remove (X) button, network error inline icon |
| Status: success | `colors.success` | `#22C55E` | Success toast bg ("게시물이 등록되었습니다", "댓글이 등록되었습니다") |
| Text primary | `colors.textPrimary` | `#1A1A2E` | Post title, post content, comment content, screen titles |
| Text secondary | `colors.textSecondary` | `#6B7280` | Card content snippet, poll option text non-active, recent search title |
| Text tertiary | `colors.textTertiary` | `#9CA3AF` | Timestamps, view-count icons, placeholder text, dismiss icons, "삭제된 댓글입니다" italic |

### Accent (`colors.primary`) — reserved for ONLY:

1. **Primary CTAs:** "등록" submit button text in `CommunityWriteScreen`, "다시 시도" retry button bg in EmptyState (via `colors.primary`), FAB ("✏️ 글쓰기") background, "댓글 보내기" send icon when input is non-empty.
2. **Active selection state:** Active sort filter label in sort bar (e.g. "Hot 🔥" when selected), active team tab in `TeamTabBar`, active poll duration chip bg in write screen (`durationChipActive`), liked-poll-option text color (`pollOptionVoted`).
3. **Brand badges:** Poll badge bg (`primaryAlpha8`) and text color in `CommunityPostCard`, team-name meta text in post detail metaRow, "답글 작성 중" reply indicator text.
4. **Active progress bars:** Poll result bar fill (`primaryAlpha8`).
5. **Pagination loading spinner color** (footer of FlatList during `onEndReached` fetch — NEW Phase 3 element).

**NOT allowed for accent:** body text, all icons by default, secondary buttons, card backgrounds, dividers, badge frames not in the list above.

### Destructive (`colors.error`) — reserved for ONLY:

1. **Liked heart icon** (post and comment, fill state).
2. **Trending badge** ("트렌딩") text — uses semantic alias `colors.trending` (= `#EF4444`).
3. **Error toast** background (already wired in `ToastContext`).
4. **Destructive menu items**: "삭제", "신고" text in Alert / context sheet (already convention).
5. **Char-count over-limit** indicator (existing `charCountOver` style).
6. **Image remove** (X) button on write screen image thumbnail.
7. **Inline error EmptyState icon** (when list-fetch fails) — NEW Phase 3.

**NOT allowed for destructive:** disabled buttons, dividers, default text, success states.

## Interaction Contracts

### Optimistic like (D-10)

```
User taps heart
  ↓
hapticMedium() (existing)
  ↓
Predict: likedIds.add(targetId), like_count += 1, UI re-renders SAME FRAME
  ↓
DB INSERT/DELETE community_likes
  ↓ (success)
DB trigger updates like_count → next refresh syncs (no visible delta)
  ↓ (failure)
Rollback: likedIds.delete(targetId), like_count -= 1
showToast(t('community_like_failed'), 'error')
```

The visual heart-fill animation must NOT wait for DB. The predictive state lives in `CommunityContext.likedIds` Set + a local `optimisticLikeDeltas: Record<string, number>` map that adds to `post.like_count` at render time.

### Pagination (D-05)

```
User scrolls to end of FlatList (onEndReached, threshold 0.5)
  ↓
If isLoadingMore || !hasMore → no-op
  ↓
setIsLoadingMore(true)
ListFooterComponent renders <ActivityIndicator color={colors.primary} />
  ↓
fetchPosts({ teamId, sort, range: [page*20, page*20+19] })
  ↓ (success)
Append to posts state, setPage(p+1), setIsLoadingMore(false)
If returned 0 rows → setHasMore(false), hide footer
  ↓ (failure)
setIsLoadingMore(false), showToast(t('community_load_error_desc'), 'error')
Footer hides; user can pull-to-refresh to retry
```

### Image upload + post creation (D-09)

```
User taps "등록"
  ↓
canSubmit gate (title + content + optional poll valid)
  ↓
Show UploadingOverlay (block UI)
Set caption: "이미지를 업로드하는 중..."
  ↓
For each image (sequential — current r2Upload.ts pattern):
  Update caption: "이미지 업로드 중 (i/N)"
  await uploadCommunityImages(...)
  ↓ (any failure)

  ↓ (all success)

  ↓ (failure)
Hide overlay → Alert with retry → form retained
  ↓ (success)
Hide overlay → showToast('write_post_success', 'success') → navigation.goBack()
```

### Report flow (D-14)

```
User taps "신고" in post action sheet
  ↓
If reportedIds.has(targetId) → Alert(community_already_reported) → exit
  ↓
Confirm Alert → on confirm → reportTarget(...)
  ↓
DB INSERT community_reports
  ↓ (success)
showToast(t('toast_report_submitted'), 'success')

```

## Accessibility Contract

Phase 3 inherits all existing a11y patterns. New requirements:

1. **All new interactive elements** (retry button, pagination spinner, dismiss recent-search) MUST have `accessibilityLabel` from the i18n table above.
2. **Optimistic like state** — `accessibilityState={{ selected: isLiked }}` plus the toggle label switches between `t('a11y_like_button')` and `t('a11y_unlike_button')`.
3. **Disabled poll options** (expired) MUST set `accessibilityState={{ disabled: true }}` and use the disabled-label key.
4. **Loading overlay** during image upload MUST set `accessibilityViewIsModal={true}` and `importantForAccessibility="yes"` so VoiceOver focus stays on the overlay caption.
5. **Skeleton placeholders** MUST set `importantForAccessibility="no-hide-descendants"` so screen readers do not announce shimmer noise.
6. **Touch targets** for retry button, dismiss icons, image-remove X — minimum 44pt hit area (use `hitSlop` if visual is smaller).
7. **Color contrast** — every new pairing in this spec uses tokens that already meet WCAG AA in Phase 1/2 audit. No new colors introduced.
